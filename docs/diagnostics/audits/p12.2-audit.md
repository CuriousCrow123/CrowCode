- `clamp(15, 0, 10)`: val=15 > hi=10, so ternary yields hi=10. `max(0, 10)`: 0>10 is false, return b=10. **a=10 correct.**

- `clamp(5, 0, 10)`: val=5 > hi=10 is false, so ternary yields val=5. `max(0, 5)`: 0>5 is false, return b=5. **c=5 correct.**

## Bugs Found

- [ ] BUG-p12.2-1: **Clamp frame removed before max frame appears.** The instrumented clamp function calls `__crow_pop_scope()` before `return max(...)`. This generates a removeEntry for the clamp frame before max is entered. In real C, clamp's stack frame persists until max returns and clamp itself returns. The user never sees clamp and max coexisting on the stack, losing the nested call visualization.

- [ ] BUG-p12.2-2: **No steps shown inside max() body.** The `max` function has `__crow_step(1)` at entry, `__crow_step(2)` inside the `if (a>b)` branch, and an implicit fallthrough to `return b`. The step(1) ops (push_scope + decl of a,b) are flushed at step(1), but `__crow_step(2)` (when the branch is taken) flushes an empty accumulation (dropped). When the branch is NOT taken (fallthrough to `return b`), there is no step call at all for line 3. In all three max calls, the user sees the function entered (line 1) and then immediately returned from, with no visibility into the branch decision on line 2 or the return on line 3.

- [ ] BUG-p12.2-3: **No step for clamp body (Line 6).** The clamp function has `__crow_step(5)` at entry but no step call for line 6. The user sees clamp's parameters (line 5) but never sees line 6 highlighted for the ternary + return expression. The pop_scope happens immediately after step(5).

- [ ] BUG-p12.2-4: **Step 10 tagged Line 11 instead of Line 12.** The `return 0;` on source line 12 has no dedicated step. The main scope cleanup is recorded under the previous step(11) tag, so the user sees line 11 highlighted twice instead of seeing line 12 for the final return.

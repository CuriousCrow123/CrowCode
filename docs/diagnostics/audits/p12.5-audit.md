### Note on base case steps

Every base case return (n<=0 or n==1) executes a `__crow_step()` that flushes an empty op accumulation (the entry ops were already flushed by the preceding `__crow_step(1)`). These empty steps are dropped. The `__crow_pop_scope()` removeEntry op gets accumulated and flushed at the *next* function's `__crow_step(1)` or at main's `__crow_step()`. This means base case decisions (lines 2 and 3) are never visible to the user.

## Bugs Found

- [ ] BUG-p12.5-1: **Recursive frames not stacked.** Same as p6.4. Each recursive call removes the previous fib frame before pushing a new one (due to `__crow_pop_scope()` before `return fib(n-1) + fib(n-2)`). The user only ever sees one fib frame at a time. For fib(6) with its deep recursion tree, the user completely loses sight of the call stack depth and the tree structure. This is the most damaging manifestation of the frame-stacking bug since fibonacci has branching recursion.

- [ ] BUG-p12.5-2: **Base case steps (Lines 2 and 3) never shown.** Across all 25 recursive fib calls in fib(6), every base case (n<=0 returning 0, n==1 returning 1) has its `__crow_step(2)` or `__crow_step(3)` dropped because it flushes an empty accumulation. The user never sees line 2 or line 3 highlighted. The only visible line inside fib is line 1 (function entry), repeated for every call.

- [ ] BUG-p12.5-3: **Step 31 tagged Line 9 instead of Line 10.** The `return 0;` on source line 10 has no dedicated step. The main scope cleanup is recorded under the step(9) tag, so line 9 is highlighted twice instead of showing line 10 for the final return.

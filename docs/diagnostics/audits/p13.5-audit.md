### Value verification details:

**Line 5 — `a = b = c = 42;`:**
In C, chained assignment evaluates right-to-left: c=42, then b=42, then a=42. The instrumented source only calls `__crow_set("a", &a, 5)` — it does not instrument b or c. The WASM runtime correctly computes all three assignments (confirmed by line 6 where c+8=50, meaning c was actually set to 42 at runtime). But the dump only shows setValue for `a`. The snapshot confirms b=0, c=0 after step 4 — meaning the instrumentation failed to emit ops for b and c.

**Line 6 — `a = b = c + 8;`:**
At runtime, c=42 (from line 5), so c+8=50. Then b=50, a=50. The instrumentation only emits setValue for `a`. The value a=50 is correct (confirming c was indeed 42 at runtime), but b remains showing 0 in the snapshot. The actual runtime value of b is 50 but the snapshot shows 0.

## Bugs Found

- [ ] BUG-p13.5-1: Chained assignment `a = b = c = 42` only instruments the leftmost target `a`. The intermediate assignments to `b` and `c` are not captured — no `__crow_set` calls are emitted for them. Snapshot shows b=0, c=0 after this line when all three should be 42.
- [ ] BUG-p13.5-2: Chained assignment `a = b = c + 8` only instruments the leftmost target `a`. The assignment to `b` is not captured. Snapshot shows b=0 after this line when b should be 50.

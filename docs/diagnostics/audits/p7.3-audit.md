## Bugs Found

- [ ] BUG-p7.3-1: **Redundant intermediate setValue ops on loop increment.** Every for-loop increment step produces two setValue ops for the loop variable: one with the pre-increment value (from the explicit `__crow_set` call before `j++`/`i++`) and one with the post-increment value (auto-detected at step boundary). The pre-increment value is stale and redundant. Occurs at Steps 5, 7, 9, 12, 14, 16, 19, 21, 23. While the final snapshot value is correct (post-increment wins), the intermediate op could confuse op-level consumers.

- [ ] BUG-p7.3-2: **Loop variable terminal value not captured when loop exits.** When the inner for-loop condition fails (j reaches 3) and the loop exits, the final j=3 value is never recorded. The snapshot shows j=2 (the last value captured by `__crow_set` before `j++`). Same issue with i at program end: i=2 shown instead of i=3. The `__crow_set` fires before the increment, and the auto-detection at the next step boundary captures the *next* iteration's state rather than the terminal value. Occurs at Steps 9, 16, and 23.

- [ ] BUG-p7.3-3: **Final step reports wrong source line.** Step 23 reports Line 5 (from `__crow_step(5)`, the `total += 1` line). The cleanup (removeEntry) semantically corresponds to `return 0` on line 8 or the end of the outer for-loop on line 7. The line attribution is misleading because the last `__crow_step` happened to be inside the loop body.

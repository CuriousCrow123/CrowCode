### Notes on step 7 line number

Step 7 is tagged Line 6. The instrumented code has `__crow_step(6)` after `__crow_decl("result"...)`, then `__crow_pop_scope()` before `return 0`. The pop_scope ops are flushed under the step(6) tag since no subsequent `__crow_step()` re-tags them. Line 7 (`return 0`) has no dedicated step.

## Bugs Found

- [ ] BUG-p6.4-1: **Recursive frames not stacked.** The instrumented code calls `__crow_pop_scope()` before `return n * factorial(n-1)`, which generates a `removeEntry` for the current frame before the recursive call pushes a new frame. In real C execution, all recursive stack frames exist simultaneously. The user only ever sees one factorial frame at a time (each step removes the previous before adding the next), losing the key visualization of recursion depth. This is caused by the instrumentation pattern placing `__crow_pop_scope()` before the recursive `return` expression.

- [ ] BUG-p6.4-2: **Base case step (Line 2) dropped.** When `n=1`, the code executes `__crow_step(2); __crow_pop_scope(); return 1;`. The `__crow_step(2)` flushes the current accumulation (which is empty -- step(1) just flushed the push_scope + decl ops). Since the step has zero ops, it is dropped/invisible. The `__crow_pop_scope()` removeEntry op is then accumulated and flushed at the next `__crow_step(6)` in main. The user never sees line 2 highlighted for the base case decision.

- [ ] BUG-p6.4-3: **Step 7 tagged Line 6 instead of Line 7.** The `return 0;` on source line 7 has no step. The main scope cleanup (removeEntry) is recorded under step 7 tagged as Line 6 (the previous `__crow_step(6)` tag carries over). The user sees line 6 highlighted twice in a row instead of seeing line 7 for the final return.

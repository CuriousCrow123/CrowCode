### Value verification details:
- arr after init: {10,20,30,40,50}. Correct.
- arr[0] = 99: {99,20,30,40,50}. Correct.
- arr[4] = 99+20 = 119: {99,20,30,40,119}. Correct.
- sum accumulation: 0+99=99, +20=119, +30=149, +40=189, +119=308. All correct.
- Note: The __crow_set for `i` in the loop emits the old value then the new value (two setValue ops per increment). This is redundant but not incorrect since the final value is right.

**Step 15 issue:** When the loop exits after i=4, the for loop should do i++ making i=5, then the condition i<5 fails. The dump shows setValue(i,4) and removeEntry(main). The final value of i should be 5 (the value that caused the loop to exit), but the snapshot never shows i=5. The step emits setValue(i,4) which is a no-op (i was already 4), then removes main. It appears the final i++ is either not executed or the __crow_set fires before the increment.

Also note: Step 15 is labeled Line 7 but should be the loop-exit step at Line 6. The line attribution is slightly off.

## Bugs Found

- [ ] BUG-p3.1-1: On loop exit, final i++ is not reflected. After the last iteration (i=4), the for loop should increment i to 5 before the condition check fails. The dump shows i remaining at 4 at scope cleanup. The `__crow_set("i", &i, 6)` at end of loop body fires with the pre-increment value, and the post-increment to 5 is never captured.
- [ ] BUG-p3.1-2: Step 15 is attributed to Line 7 (the loop body) but should be attributed to Line 6 (the for statement) or Line 9 (return 0), since this step represents loop exit and scope cleanup.

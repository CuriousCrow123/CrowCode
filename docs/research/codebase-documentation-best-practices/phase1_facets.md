# Phase 1: Facet Decomposition

## Topic
Best practices for codebase documentation: what to document, where to put it, how to keep it maintainable.

## Facets

### 1. Inline Documentation
**Question:** What are current best practices for inline comments, docstrings, and self-documenting code — when to comment vs. when code should speak for itself?
**Search seeds:** "inline code comments best practices 2025", "when to write code comments vs self-documenting code", "docstring conventions best practices"

### 2. Project-Level Documentation
**Question:** What should README files, architecture docs, and ADRs contain, and how should they be structured for maximum usefulness?
**Search seeds:** "architecture decision records best practices", "README best practices software projects", "architecture documentation techniques arc42 c4"

### 3. API Documentation
**Question:** What are effective approaches to API documentation — auto-generated vs hand-written, tools, and keeping them in sync with code?
**Search seeds:** "API documentation best practices 2025", "auto-generated documentation tools comparison", "docs as code API documentation"

### 4. Developer Onboarding Documentation
**Question:** What documentation best supports new developer onboarding, and what formats/structures are most effective?
**Search seeds:** "developer onboarding documentation best practices", "contribution guide template open source", "new developer getting started guide"

### 5. Documentation Maintenance
**Question:** How do teams keep documentation accurate over time — docs-as-code, CI checks, ownership models, and anti-patterns?
**Search seeds:** "keeping documentation up to date engineering teams", "docs as code workflow 2025", "documentation rot prevention strategies"

## Known Tensions
- Self-documenting code vs. comments — how much is enough?
- Auto-generated docs vs. hand-written prose
- Comprehensive documentation vs. maintainability burden
- Centralized wiki vs. docs-in-repo (docs-as-code)
- Documentation for current team vs. future readers

## Likely Gaps
- Empirical data on documentation ROI
- Metrics for documentation quality
- Small team vs. large organization differences

## Recency Sensitivity
Moderate — tooling evolves but core principles are stable. Bias toward recent sources for tooling recommendations.
